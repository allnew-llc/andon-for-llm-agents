**Structure (in this order):**

   | Quality self-assessment | Evaluating deliverable quality against criteria | (no args) |

   Key signals: "quality", "assessment", "audit", "gate score", "pass rate", "trend", "defect rate", "deliverable quality"

3. `## Usage` section:
   ```
   /qc-audit                  Execute quality self-assessment for current phase
   /qc-audit trend            Display quality trend over time (FAIL/WARN/OK rates)
   /qc-audit gate-health      Show correlation between quality scores and gate pass rates
   ```

4. `## Subcommand: (no arguments)` section — This is the CORE instruction-based skill (QC-02). Write detailed step-by-step instructions telling Claude HOW to perform a quality self-assessment:

   **Step 1: Identify Current Phase**
   - Determine the current phase from project context (ROADMAP.md, STATE.md, or user context)
   - Locate the phase's deliverable manifest or success criteria

   **Step 2: Load Quality Criteria**
   - Read the deliverable manifest for the current phase (if using ios-app-factory, check the pipeline config; otherwise check ROADMAP.md success criteria)
   - For each deliverable, extract `quality_criteria` items
   - If no explicit manifest exists, derive criteria from the phase's success criteria in ROADMAP.md

   **Step 3: Assess Each Criterion**
   - For each quality criterion, examine the actual deliverable artifact
   - Assign status: `pass` (criterion clearly met with evidence), `warn` (partially met or evidence is thin), `fail` (criterion not met or artifact missing/empty)
   - Record the evidence (file examined, what was found, why this status)

   **Step 4: Compute Overall Status**
   - `pass`: All criteria are pass
   - `warn`: No fails, but at least one warn
   - `fail`: At least one criterion is fail

   **Step 5: Output Results**
   - Display a table: | Criterion | Status | Evidence |
   - Show overall: PASS / WARN / FAIL
   - Save to `docs/pipeline/quality-self-assessment-{phase_id}.json` with structure:
     ```json
     {
       "phase_id": "<current_phase>",
       "timestamp": "<ISO8601>",
       "overall": "pass|warn|fail",
       "criteria": [
         { "name": "<criterion>", "status": "pass|warn|fail", "evidence": "<what was checked>" }
       ]
     }
     ```

   **Step 6: Recommend Actions**
   - If FAIL: List specific fixes needed before gate submission
   - If WARN: List improvements to strengthen the deliverable
   - If PASS: Note readiness for gate submission
   - Reference: "Do NOT submit to gate with FAIL items — fix first (see rules/45-quality-driven-execution.md)"

5. `## Subcommand: trend` section:
   - Purpose: Display historical quality trend from `${CLAUDE_PLUGIN_DATA}/qc/` or project-local `docs/pipeline/`
   - Action: Run `skills/qc-audit/scripts/trend.sh` (created in Plan 02)
   - Interpretation guide: "improving" = quality increasing, "degrading" = investigate causes, "stable" = monitor
   - Tip: Run after closing ANDON or completing kaizen cycle to measure impact

6. `## Subcommand: gate-health` section:
   - Purpose: Analyze correlation between quality self-assessment scores and gate pass/fail rates
   - Action: Run `skills/qc-audit/scripts/gate-health.sh` (created in Plan 02)
   - Interpretation guide: Explain what "high quality score but gate failure" means (gate conditions misaligned with quality), and what "low quality score but gate pass" means (gate conditions too lenient)

7. `## Gotchas` section with 5+ named failure patterns:

   ### Score Inflation
   Agent inflates pass/warn assessments to avoid fixing deliverables. Evidence column contains vague justifications ("looks good", "seems complete") instead of specific file/line references. Prevention: Evidence must cite specific files, line counts, or measurable attributes.

   ### Criteria Drift
   Quality criteria are modified or reinterpreted to match existing deliverables rather than improving deliverables to match criteria. The assessment bends the ruler to fit the measurement. Prevention: Load criteria from source (manifest/ROADMAP) before examining deliverables — never derive criteria from what already exists.

   ### Trend Blindness
   Running /qc-audit only for the current phase without checking historical trends. A single PASS hides a degrading trajectory. Prevention: Run `trend` periodically, not just before gate submission.

   ### Gate-Quality Conflation
   Treating gate pass as equivalent to quality pass. Gates check minimum conditions (file exists, markers present). Quality assessment checks whether the deliverable is genuinely useful. A deliverable can pass the gate and still be low quality. Prevention: Always run /qc-audit assessment before gate submission, not after.

   ### Assessment-Only Syndrome
   Running quality self-assessment but not acting on WARN/FAIL results. The assessment becomes a checkbox exercise. Prevention: FAIL items MUST be fixed before gate submission. WARN items should be addressed unless there is a documented reason to defer.

8. `## Related Skills` section:
   | Skill | Path | When to Chain |
   |-------|------|---------------|
   | tps-kaizen | skills/tps-kaizen/SKILL.md | After ANDON close — run /qc-audit to verify quality is restored |
   | tps-kaizen audit | skills/tps-kaizen/SKILL.md | Before /qc-audit — use 3M audit to identify waste patterns affecting quality |

   ### Composition Patterns
   **Quality Gate Chain**: `qc-audit` (self-assess) -> fix FAIL items -> `qc-audit` (re-assess) -> gate submission
   **Post-Incident Chain**: `tps-kaizen andon` (stop + fix) -> `tps-kaizen five-whys` (root cause) -> `qc-audit` (verify quality restored)
   **Continuous Improvement Chain**: `qc-audit trend` (identify trajectory) -> `tps-kaizen kaizen` (plan improvement) -> `qc-audit gate-health` (measure impact)
