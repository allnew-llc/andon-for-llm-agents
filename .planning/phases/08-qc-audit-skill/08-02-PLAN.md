JSON extraction helper (awk, no jq):
```bash
extract_json_str() {
    local key="$1"
    local file="$2"
    awk -v key="$key" '
    {
        pattern = "\"" key "\"[[:space:]]*:[[:space:]]*\"([^\"]*)\""
        if (match($0, pattern)) {
            s = substr($0, RSTART, RLENGTH)
            sub("^\"" key "\"[[:space:]]*:[[:space:]]*\"", "", s)
            sub("\"$", "", s)
            print s
            exit
        }
    }
    ' "$file"
}
```

Status counting (awk gsub, not grep|wc -l):
```bash
count_status() {
    local val="$1"
    local file="$2"
    awk -v val="$val" '{
        n += gsub("\"status\"[[:space:]]*:[[:space:]]*\"" val "\"", "")
    } END { print n+0 }' "$file"
}
```

  <files>skills/qc-audit/scripts/gate-health.sh</files>
  <action>
Create `skills/qc-audit/scripts/gate-health.sh` — a portable bash script that analyzes the correlation between quality self-assessment scores and gate pass/fail rates.

**Follow the same established patterns** as Task 1 (license header, set -euo pipefail, usage(), env var resolution, awk JSON extraction).

**Data sources:**
1. Quality self-assessments: `${CLAUDE_PLUGIN_DATA}/qc/quality-self-assessment-*.json` (or fallback `docs/pipeline/`)
2. Gate results: `${CLAUDE_PLUGIN_DATA}/qc/gate-results/gate-result-*.json` (or fallback `docs/pipeline/gate-result-*.json`)

**Logic:**
1. For each phase_id, find the quality assessment overall status and the gate result
2. Join on phase_id to build a correlation table
3. Categorize each phase into one of four quadrants:
   - PASS-PASS (quality pass + gate pass) = "Healthy"
   - PASS-FAIL (quality pass + gate fail) = "Gate Misaligned" — gate catches something quality missed
   - FAIL-PASS (quality fail + gate pass) = "Gate Too Lenient" — gate passed despite poor quality
   - FAIL-FAIL (quality fail + gate fail) = "Correctly Caught" — system working as intended
   - (WARN counts as non-PASS for the quality side)
4. Print the correlation matrix and interpretation

**Output format:**
```
=== Gate Health Report ===
Data source: /path/to/qc/
Phases analyzed: N

Phase           | Quality | Gate   | Diagnosis
----------------+---------+--------+------------------
1a              | PASS    | PASS   | Healthy
2a              | WARN    | PASS   | Gate Too Lenient
2b              | FAIL    | FAIL   | Correctly Caught
3               | PASS    | FAIL   | Gate Misaligned

Correlation Matrix:
                  Gate PASS    Gate FAIL
  Quality PASS      2            1
  Quality WARN/FAIL 1            1

Health Score: N/M phases healthy (X%)

Interpretation:
  - "Gate Too Lenient" phases: Gate conditions may need strengthening
  - "Gate Misaligned" phases: Quality criteria may be incomplete
  - High health score (>80%): Quality and gates are well-aligned

=== End of Report ===
```

If gate result files don't exist (common case for projects not using pipeline gates), print:
```
No gate result data found in ${DATA_DIR}/gate-results/
Tip: Gate results are generated by pipeline gate checkers. This analysis requires both quality assessments AND gate results.
```
and exit 0.

Create `skills/qc-audit/scripts/collect-assessments.sh` — a portable bash script that aggregates quality-self-assessment JSON files from multiple project directories into the central `${CLAUDE_PLUGIN_DATA}/qc/` store.

**Follow the same established patterns** as Tasks 1-2 (license header, set -euo pipefail, usage(), awk JSON extraction).

**Usage:**
```
collect-assessments.sh [PROJECT_DIR ...] [--help]
```

If no PROJECT_DIR arguments are given, scan the current working directory.
If arguments are given, scan each provided directory.

**Logic:**
1. Resolve target store: `${CLAUDE_PLUGIN_DATA}/qc/` (MUST be set for this script — it aggregates TO the central store)
   - If CLAUDE_PLUGIN_DATA is not set, print error: "CLAUDE_PLUGIN_DATA must be set for collection. This script aggregates project-local assessments into the central store." and exit 1.
2. Create `${CLAUDE_PLUGIN_DATA}/qc/` if it doesn't exist (`mkdir -p`)
3. For each project directory:
   - Search for `docs/pipeline/quality-self-assessment-*.json` files
   - For each found file:
     - Extract phase_id from the JSON (or from filename)
     - Check if same phase_id already exists in central store (skip if newer version exists, overwrite if source is newer)
     - Copy file to `${CLAUDE_PLUGIN_DATA}/qc/` preserving the filename
     - Print what was collected: `[collected] project/docs/pipeline/quality-self-assessment-1a.json -> ${CLAUDE_PLUGIN_DATA}/qc/`
4. Print summary: "Collected N assessments from M project(s)"

**Edge cases:**
- Project dir doesn't exist: warn and skip
- No assessment files found in a project: report "(0 assessments found)" and continue
- File already exists in central store with same content: skip with `[skipped]` (already up-to-date)
- Use file mtime comparison (macOS `stat -f '%m'` for epoch seconds) to determine "newer"

**Output format:**
```
=== Collecting Quality Assessments ===

=== Done ===
```
